Containerized test strategy
===========================

Purpose
-------

This note is an action plan, not a raw wishlist. It turns the requirement
"containerized tests only" into a staged engineering program for this repo's
actual risk: an operator toolkit that installs cleanup policy and therefore
must not let tests normalize host mutation, hidden host dependencies, or unsafe
cleanup behavior.

The plan deliberately separates:

- containment boundary
- harness posture
- functional installer/uninstaller contracts
- cleanup-policy semantics
- distro compatibility
- optional CI

Do not treat the checklist as permission to make broad mechanical churn. Each
stage should make the repository more coherent on its own.

Context
-------

This repository is an operator toolkit for installing small VPS cleanup
policies. The risk profile is different from an application repository:
incorrect tests can accidentally normalize host mutation, sudo habits, Docker
socket access, or partial installs. The test harness therefore needs to be part
of the safety contract, not a convenience wrapper around host scripts.

Limits of confidence
--------------------

Containerized tests can prove the repository's installer logic, file rendering,
policy detection, refusal behavior, and cleanup-command construction under
known Debian/Ubuntu userlands. They cannot fully prove that a real VPS service
manager, Docker daemon, logrotate timer, or apt timer is configured correctly
after a privileged install. That final operational verification should remain a
small documented post-install check, not something hidden inside the automated
test suite.

The test suite should therefore avoid pretending to be a live VPS. It should
prove that the repo will install the right artifacts safely and that the
operator-facing commands are bounded and explicit.

Threat model
------------

The test system is designed to defend against:

- accidental host mutation while developing tests
- tests passing only because the host happens to have certain tools installed
- tests writing artifacts into the checkout
- tests receiving host credentials, sockets, or service access
- Docker cleanup tests accidentally reaching a real Docker daemon
- installer tests normalizing partial writes or unsafe symlink behavior
- private/local state leaking into the test image through build context
- future maintainers weakening containment flags without noticing

The test system is not designed to defend against a malicious maintainer who
edits both the harness and the tests to hide bad behavior. Posture tests raise
the cost of accidental weakening; they are not a substitute for code review.

Hard requirements
-----------------

1. Tests are container-only.
   Direct host execution of test entrypoints should fail closed. The supported
   host action is only to build/run the test container.

2. The repository must not be writable from tests.
   The test runtime should not bind-mount the checkout read-write. The default
   should be a copied-context test image so the running tests do not touch the
   host checkout at all. If an interactive debug lane uses a bind mount, it must
   be read-only.

3. The test runtime must not touch host services or host state.
   No host Docker socket, no sudo, no systemctl against the host, no host /tmp
   writes, no host repo writes, no host network during test execution. The
   unavoidable exception is outer container-engine state: building and running
   an image necessarily uses the host Docker/Podman image store. The tests
   themselves must not receive a path, socket, credential, or network route
   that lets them mutate the host.

4. Image inputs should be pinned intelligently.
   Runtime base images should be named-release plus digest-pinned, for example
   Debian 12/bookworm and Ubuntu 24.04/noble. Do not use floating "stable" or
   "latest". Apt packages do not need exact version pins initially because
   distro tool drift is useful signal for this repo; if CI becomes flaky, move
   the test image to Debian/Ubuntu snapshot repos as a deliberate second step.

5. Do not use Compose unless it buys something real.
   This repo has no database, sidecar, internal network, or multi-service test
   topology. Direct docker run is simpler and easier to audit. Compose is a
   rejected default here. Keep DOCKER_CLI overridable, but do not claim Podman
   support until the runner has actually been tested with Podman.

Non-negotiable invariants
-------------------------

These are stronger than ordinary checklist items. If any invariant is broken,
the test harness is not acceptable even if the functional assertions pass.

- Functional tests must not run directly on the host.
- Test containers must not receive the host Docker socket.
- Test containers must not receive writable host path mounts.
- The default verification lane must not receive any repository bind mount.
- Test execution must not have runtime network access.
- Test execution must not receive host credentials or host HOME.
- The broad test lane must run as non-root.
- Any root lane must be separate, narrow, and metadata-focused.
- Docker cleanup behavior must be tested only with fake Docker commands.
- README verification instructions must point to the containerized entrypoint.

Decision frame
--------------

Prefer work that removes ambiguity from a boundary.

Do not begin by splitting every test file, adding CI, or polishing the matrix.
Those are useful only after the core host/test boundary is mechanically true.

Use this ordering rule:

1. Make host execution impossible for test entrypoints.
2. Make the default test container maximally contained.
3. Make the harness posture observable and testable.
4. Make the installer/uninstaller contracts precise.
5. Make cleanup-policy semantics explicit.
6. Then broaden to distro matrix, root metadata, and CI.

Before implementing any item, re-check the current code. This note is a plan,
not authority over reality. A listed item can be stale, partially covered, or
right about the smell with the wrong implementation shape.

How to use this plan
--------------------

Work by iteration. Take the smallest current pressure, inspect the code, write
the smallest truthful change, read it back, remove unearned structure, then
check the adjacent boundaries it touched: tests, README, fixtures, wrapper
posture, cleanup behavior, and failure output.

Small slices are mandatory. A slice should be small enough that its boundary,
risk, tests, and rollback story are obvious. If a change wants to touch the
container runner, test splitting, README, fake Docker fixtures, and CI at once,
it is too large. Finish the smallest coherent slice, read it back, and let that
result teach the next slice.

Prefer this loop:

1. Pick one boundary.
2. State the current smell in one sentence.
3. Inspect the exact files that own it.
4. Add or adjust one narrow check if behavior is observable.
5. Make the smallest change that satisfies that check.
6. Re-read the changed files and remove anything speculative.
7. Run the narrow verification.
8. Run the adjacent-boundary verification.
9. Update the note/checklist if the work changed the plan.

For every implementation slice:

- [ ] I pledge to re-read the non-negotiable invariants before editing, inspect
      the real code before trusting this plan, make the smallest truthful
      change, read it back, delete unearned structure, check adjacent
      boundaries, and stop only when the relevant "Done means" statement is
      true.
- [ ] State the boundary being improved.
- [ ] Verify the current behavior in code before editing.
- [ ] Prefer a narrow failing check first when behavior is observable.
- [ ] Make the smallest change that eliminates the class of bug.
- [ ] Read the change back and remove unearned abstraction.
- [ ] Check adjacent boundaries before moving on.
- [ ] Update docs when the operator-facing command changes.
- [ ] Keep each commit to one coherent concern.

Do not mark a stage complete merely because files changed. Mark it complete
only when the stage's "Done means" statement is true.

Assumption audit
----------------

Assumption: "Container-only" means the host may do nothing except run Docker.

Decision: correct, but state the boundary precisely. The host wrapper may read
the checkout as build context and may create image/layer/container metadata in
the host container engine. It must not run repository tests, install package
dependencies, write test artifacts into the checkout, mount host service
sockets, or invoke host cleanup/systemd commands.

Assumption: a read-only bind mount is enough.

Decision: not for the default lane. A read-only bind mount prevents writes to
the checkout, but it still gives the test process a live view of host files.
The default lane should build an image from the checkout and run tests from the
copied image filesystem. A debug shell may use a read-only bind mount only if
it is clearly not the default verification lane.

Assumption: digest-pinning the base image makes builds reproducible.

Decision: only partially. Digest-pinning controls the starting filesystem, but
apt package resolution still changes with repository state. That is acceptable
at first because this project cares about current Debian/Ubuntu tool behavior.
If reproducibility becomes more important than drift detection, switch to
snapshot repositories and record the snapshot date.

Assumption: noexec /tmp is easy.

Decision: false with the current tests. They create executable fake Docker
commands in tmp. The harness should not weaken /tmp for that convenience.
Refactor fake commands into checked-in fixtures or invoke generated scripts via
the shell so tmpfs can be noexec.

Assumption: a root-in-container lane is dangerous.

Decision: root is acceptable only as a narrow model of sudo-installed file
metadata. It must not get network, user-provided host path mounts, the Docker
socket, broad capabilities, or privileged mode. If ownership tests need
CAP_CHOWN, grant only that capability to that lane and test that the broad lane
does not have it.

Best-practice synthesis
-----------------------

The local repo survey and Docker's own documentation point to the same shape:

- Use small, trusted base images and avoid unnecessary packages. This keeps the
  test image understandable and reduces unrelated vulnerability/noise surface.
- Keep Make as the stable host command surface. Neighboring repos consistently
  use Make targets as the operator interface even when the actual containment
  logic lives in a shell wrapper or Compose file.
- Prefer a direct `docker run` wrapper when the lane is single-container.
  Local repos that need only a toolchain or test runner use small wrappers or
  Make recipes; repos that use Compose do so because they have an internal
  service topology such as PostgreSQL, Haraka, or app-sidecars. This repo has
  no such topology.
- Pin base images by named release plus digest. Docker documents that tags are
  mutable and digest pins preserve the exact base image. The tradeoff is that
  digest pinning opts out of automatic base-image fixes, so the plan also needs
  an explicit update procedure.
- Use .dockerignore deliberately. Docker removes ignored files from the build
  context before sending it to the builder; that is exactly the control needed
  to keep .git, agent metadata, caches, credentials, and local artifacts out of
  the test image.
- Prefer ephemeral containers. Tests should leave no container-local state
  except the outer Docker image/cache metadata needed to run them.
- Use --read-only only with explicit writable tmpfs mounts. Docker's run docs
  describe read-only rootfs as prohibiting writes except to specified writable
  mounts; this makes tmpfs locations part of the test contract.
- Never mount the host Docker socket. Docker's run docs are clear that a
  container with the Docker socket can create and manipulate the host daemon.
  Fake Docker command fixtures are the right test double for this repo.
- Use explicit runtime limits and privilege reduction: no-new-privileges,
  cap-drop ALL, no privileged mode, no runtime network, PID/memory limits, and
  a non-root default user.
- Add posture tests for the harness text and for the in-container environment.
  This is a common useful pattern in nearby repos: text checks catch accidental
  removal of critical flags, while runtime checks prove the container actually
  started with the intended boundary.
- Use a marker guard for directly executable test entrypoints. The strongest
  nearby host-refusal example routes public commands through a container
  dispatcher and tests that the dispatcher refuses host execution; this repo
  should make that behavior explicit for test scripts.
- Refuse root host invocation for Docker test lanes. Running the host wrapper
  as uid 0 hides permission mistakes and weakens the broad non-root contract.

Local precedent audit
---------------------

The survey of neighboring repositories changes the plan only by making several
decisions firmer:

- `the-dark-side` and `id-overlay` are useful direct-runner references: small
  Bash wrappers, array-based Docker invocation, explicit env passing, resource
  limits, read-only rootfs, dropped capabilities, no-new-privileges, and
  networkless ordinary checks.
- `nmr-api-v1` and `grimace-py` are the strongest copied-context precedents.
  Their broad test lanes build from the repository context and run from image
  state rather than a live writable host checkout.
- `agent-mail` and `nmr-api-v1` justify Compose only when tests need a private
  service topology. `auto-cleanup` should not introduce Compose merely to run a
  single shell test runner.
- `bolt-spec-v0`, `id-overlay`, and `the-dark-side` show strict containment
  around bind-mounted checkouts, but those are app/toolchain repos whose normal
  development loop needs live source access. They are not precedent for a
  bind-mounted default verification lane in this installer repo.
- `the-dark-side`, `nmr-api-v1`, and `grimace-py` all reinforce posture tests as
  first-class tests rather than README-only promises.

The consequence for this repo is: keep the default lane stricter than most
nearby app repos. Use copied image context, no runtime repository mount, no
runtime network, no Docker socket, and non-root execution as the normal
verification path. A debug shell may exist, but it must be named and documented
as a separate lane.

Base-image update policy
------------------------

Digest pins must not become stale by accident. Treat base-image refresh as a
normal maintenance change:

1. Update the named-release digest deliberately.
2. Rebuild the test image from scratch.
3. Run the full matrix.
4. Review changes in apt/systemd/logrotate behavior as signal, not nuisance.
5. Commit the digest change with any required fixture/test adjustments.

Do not use floating tags to get silent updates. Silent updates make failures
harder to explain and weaken the audit trail. If security scanning is added
later, it should report stale pinned bases and drive explicit updates rather
than mutating the base behind the suite.

Chosen shape
------------

Use a direct container runner:

    make test
    make test/matrix
    make test/root-container
    make test/posture
    make shell

The Makefile should be the operator interface and should delegate to a small
scripts/test-container.sh wrapper. The wrapper is easier to posture-test than a
large Make recipe, while Make remains the discoverable command surface.

The wrapper builds a test image from the current repository and runs tests from
the image filesystem. Runtime uses no repository bind mount by default.

Image build may use network to fetch distro packages. Test execution must use
--network none. If fully offline builds become a requirement later, publish and
pin the test image separately rather than weakening runtime isolation.

Use `docker run --pull=never` for test execution. The image should have been
built locally by the wrapper; runtime must not unexpectedly pull or resolve
anything from a registry.

Default test runtime posture:

- --network none
- --read-only
- --cap-drop ALL
- --security-opt no-new-privileges:true
- non-root numeric user, preferably 65532:65532 in the default lane
- no Docker socket
- no privileged mode
- CPU, PID, memory, and memory-swap limits
- HOME under /tmp
- PYTHONDONTWRITEBYTECODE=1
- tmpfs /tmp with nosuid,nodev,noexec

The noexec /tmp requirement matters. Current tests generate fake Docker
executables under /tmp. That should be refactored before locking /tmp down:
checked-in fake Docker command fixtures or a shell-dispatched fake command are
better than executable files generated in writable tmp.

The copied image context should include only files required by the tests.
.dockerignore is therefore part of the test contract, not cleanup polish.

The test Containerfile should use `COPY . /work` after a strict .dockerignore,
not a runtime bind mount. Selective COPY lists are tempting, but they can hide
packaging mistakes. For an installable operator repo, testing the ignored build
context is closer to testing what a clone contains.

The image build should not execute repository code. Build steps may install OS
packages and copy the checkout, but tests should run only at container runtime
under the locked runtime profile. This keeps build-time network/package access
separate from test execution.

Host wrapper responsibilities
-----------------------------

scripts/test-container.sh should be intentionally small and testable:

- refuse to run as host uid 0 unless a narrowly named override is set
- require the selected base image to come from an allowlisted matrix entry
- reject matrix image refs that are not digest-pinned
- build the image with an explicit tag derived from the matrix entry
- run tests with --pull=never
- pass only explicit environment variables into the container
- never pass SSH agent, cloud credentials, Docker config, or host HOME
- never use --env-file from the host
- never mount /var/run/docker.sock
- never mount the repository in the default test lane
- print the exact image/matrix lane being run
- clean up containers with --rm, but do not prune images/caches automatically

The wrapper itself should have a posture test that reads it as text and also
exercises a dry-run or `--print-run-command` mode. Text inspection is not
perfect, but it catches accidental removal of critical flags.

Base images
-----------

Primary image:

- Debian 12/bookworm slim, digest-pinned.
- Installs only the tools needed to test the operator contract:
  dash, python3, logrotate, systemd/systemd-analyze, coreutils, findutils,
  grep, sed, awk, util-linux if needed by the tests.

Containerfile shape:

- `ARG BASE_IMAGE=<named-release>@sha256:<digest>`
- `FROM ${BASE_IMAGE}`
- The wrapper may override BASE_IMAGE only with an allowlisted matrix value.
- The wrapper should reject BASE_IMAGE values without `@sha256:`.

Compatibility matrix:

- Debian 12/bookworm slim, digest-pinned.
- Ubuntu 24.04/noble base, digest-pinned.
- Debian 13/trixie only after it is a stable release and the test image is
  digest-pinned.

Default local test can run Debian 12 only. CI should run at least Debian 12 and
Ubuntu 24.04. Full matrix can be opt-in if build time matters.

Why not a bind-mounted repo by default?
--------------------------------------

A read-only bind mount is acceptable for an explicitly marked debug shell, but
a copied-context image is stricter for verification: the container cannot
observe host-side file changes during a run, cannot write to the checkout by
construction, and proves .dockerignore/image inputs are part of the contract.
It also makes "tests do not touch the host" easier to reason about.

The host still supplies the Docker build context. That is a read operation
performed by the outer container engine before tests begin, not a runtime mount
given to the test process.

The tradeoff is rebuild latency. For this small repo that is acceptable.
Provide make shell only for debugging, and if it mounts the repo, mount it
read-only with the same runtime restrictions.

Test entrypoint guards
----------------------

Every test script that can be executed directly should source a guard:

    AUTO_CLEANUP_TEST_CONTAINER=1

If the variable is absent, print a short error:

    tests must run inside the test container; use: make test

This is intentionally strict. It prevents local host tool versions from
becoming the unreviewed test environment.

Tests should also avoid depending only on the marker. A posture test should
verify the runner actually sets the intended container flags.

Test layout
-----------

The current tests/run.sh is valuable but too monolithic for the target
contract. Split by responsibility:

    Makefile
    scripts/test-container.sh
    containers/test/Containerfile
    containers/root-test/Containerfile, only if truly needed
    tests/run-all.sh
    tests/00_syntax.sh
    tests/01_manifest.sh
    tests/02_container_posture.sh
    tests/10_install_contract.sh
    tests/11_uninstall_contract.sh
    tests/20_docker_daemon_json.sh
    tests/21_docker_cleanup_script.sh
    tests/30_policy_detection.sh
    tests/40_docs_contract.sh
    tests/50_compat_smoke.sh
    tests/lib/assert.sh
    tests/lib/staged-root.sh
    tests/fixtures/fake-docker.sh

The exact names can change, but the split should remain contract-oriented:
installer, uninstaller, Docker daemon merge, scheduled Docker cleanup, policy
detection, docs, and harness posture.

Do not split mechanically just to reduce file length. Split where the
contract boundary is real and where failure messages become clearer.

What the tests should prove
---------------------------

Source and package integrity:

- Every manifest source exists.
- Manifest modes match actual file modes.
- Manifest destinations are absolute and unique.
- Manifest destinations match the installer/uninstaller target list.
- Runtime scripts are executable.
- Config fixtures are 0644.
- No unresolved template placeholders exist in rendered files.
- No generated bytecode, caches, backups, or temp files are present.
- .dockerignore excludes host-only state such as .git, .agents, .codex,
  caches, bytecode, backups, and temp files.
- The test image does not copy ignored host-only/private state.

Shell portability:

- sh -n and dash -n pass for all POSIX shell scripts.
- Behavioral tests run under the distro /bin/sh, not bash by accident.
- No Bash-only syntax appears in scripts that claim /bin/sh.

Installer contract:

- Default staged install creates exactly the expected file set and no extras
  outside the staged root.
- Custom --root, --prefix, and --etc-dir render correct paths.
- --root and custom --etc-dir disable service actions.
- --no-service-actions suppresses all systemctl calls.
- --no-enable-timer does not enable/start the timer.
- --restart-docker is opt-in.
- Dry-run writes nothing and prints only high-level intended actions.
- Partial checkout fails before installing anything.
- Missing fixture/helper fails before installing anything.
- Invalid existing Docker daemon JSON fails before unrelated file writes.
- --skip-docker-config removes the Python requirement and leaves daemon JSON
  untouched.
- Component skip flags install only the remaining requested policies.

Filesystem safety:

- Refuse symlink destinations.
- Refuse directory destinations.
- Refuse other non-regular destinations where the container can create them.
- Do not follow Docker daemon symlinks.
- Do not restore Docker backup symlinks.
- Reject unsafe path arguments: relative paths, /, // prefixes, whitespace,
  shell metacharacters, quotes, backticks, and dollar signs.
- Reinstall is idempotent.
- Every test-created path is under container-local tmpfs or copied image state.
- Tests should assert that staged root cleanup leaves no auto-cleanup temp
  directories behind.

Docker daemon JSON merge:

- Missing daemon file creates exact policy.
- Empty daemon file creates exact policy.
- Existing unrelated daemon keys are preserved.
- log-driver and log-opts are replaced by exact policy.
- Existing file mode is preserved.
- Existing owner/group is preserved in a focused root-in-container lane.
- Backup is created only when content changes.
- Backup content is exact original.
- Backup mode/owner are preserved where possible.
- Invalid JSON, non-object JSON, invalid UTF-8, symlink path, directory path,
  bad policy fixture, missing log opts, wrong values, and extra opts fail.
- Atomic rewrite leaves no temp clutter after failures that can be simulated.

Docker cleanup script:

- The test image should not include a real Docker CLI.
- Missing Docker CLI exits 0 with skip message.
- Unreachable Docker daemon exits 0 with skip message.
- No path can emit volume prune, system prune, --volumes, or direct
  /var/lib/docker deletion.
- Builder prune supports --keep-storage.
- Builder prune supports --reserved-space.
- Buildx fallback supports --reserved-space.
- No-storage-flag fallback prunes by age only.
- No builder/buildx support skips build cache and continues other safe prunes.
- Env overrides affect only the intended prune filters/storage reservation.
- DOCKER is treated as executable name/path, not a shell command line.
- Fake Docker commands must work with noexec /tmp; prefer checked-in fixtures
  or shell-dispatched fixtures over generated executable files.
- A failing prune command stops the script if set -e remains the intended
  behavior; otherwise change the script and test the new behavior.

Policy detection:

- btmp fixture installs when no size-based policy exists.
- btmp fixture skips only when the same active stanza includes /var/log/btmp
  and maxsize 50M.
- Commented btmp stanzas do not count.
- /var/log/btmp-old does not count.
- maxsize in another stanza does not count.
- Combined path stanzas count if logrotate accepts them.
- Newline-brace stanzas count if logrotate accepts them.
- A monthly-only distro btmp stanza does not block installing the repo policy.
- APT fixture installs when no active cleanup policy exists.
- Positive AutocleanInterval and CleanInterval count.
- "always" counts because apt.systemd.daily supports it.
- "0", update-only periodic config, and commented config do not count.
- check.sh --strict and install.sh use the same policy predicates.

systemd and logrotate:

- Rendered service has no unresolved placeholders.
- systemd-analyze verify passes in each supported image.
- Timer has OnCalendar=weekly, Persistent=true, RandomizedDelaySec=1h,
  AccuracySec=1h.
- Service has Type=oneshot, ConditionFileIsExecutable, EnvironmentFile,
  Nice=10, IOSchedulingClass=idle, TimeoutStartSec=30min.
- logrotate -d accepts the btmp fixture in every supported image.
- create 0660 root utmp is valid in supported images.

Uninstall contract:

- Removes only repo-owned installed files.
- Skip flags preserve optional policy files.
- Leaves Docker daemon config untouched by default.
- Backup restore is explicit only.
- Backup restore requires absolute safe regular file path.
- Restore destination refuses symlink and non-regular files.
- Restore preserves backup metadata as documented.
- Dry-run removes nothing.
- Service actions are disabled under --root/custom --etc-dir/no-service-actions.

Docs contract:

- README verification examples use containerized entrypoints only.
- Direct ./tests/*.sh examples should be removed or marked internal.
- All referenced repo files exist.
- Policy table matches fixtures.
- Dangerous commands appear only in the "do not automate" context.
- Docker/logrotate/journald/APT claims match fixture values and test helpers.

Container posture:

- Test runner includes no Docker socket mount.
- Test runner uses no network at runtime.
- Test runner uses read-only root filesystem.
- Test runner runs from copied image context, not a default repo bind mount.
- Test runner drops capabilities.
- Test runner uses no-new-privileges.
- Test runner uses non-root default user.
- Test runner sets memory and PID limits.
- Test runner sets tmpfs /tmp with noexec,nosuid,nodev.
- Test scripts refuse execution without the container marker.
- Any debug shell lane is clearly separate from default tests and at least as
  constrained, except for interactivity.
- The posture test should inspect the wrapper and also run an in-container
  self-check that confirms marker env, uid, writable paths, network absence,
  and absence of Docker socket.

Failure diagnostics
-------------------

The suite should be strict without becoming opaque:

- Every test file should print a short lane name before major sections.
- Assertion helpers should include expected value, actual value, and path.
- On failure, staged roots should remain inspectable inside the container only
  when an explicit debug flag is set.
- Default runs should clean all staged roots and temporary files.
- The wrapper should print the exact matrix lane and local image tag.
- The wrapper should not print secrets or inherited host environment.
- The debug shell should use the same image and posture, with any deviations
  printed clearly.

Root-in-container lane
----------------------

Most tests should run as non-root. A small root-in-container lane is justified
only for behavior that models sudo-installed metadata:

- root-owned Docker daemon JSON preservation
- backup owner/group preservation
- root/utmp logrotate assumptions if necessary

This lane must still have:

- no network
- no user-provided host path mounts
- read-only rootfs
- tmpfs /tmp
- no Docker socket
- no privileged mode
- copied image context, not a bind-mounted checkout

Start with uid 0 and cap_drop ALL. If the metadata tests prove CAP_CHOWN is
required to exercise owner transitions, add only CAP_CHOWN to this lane and
document exactly which assertion needs it. Do not give the broad suite root or
extra capabilities.

Be precise about what "root" means here. A real sudo install normally has
capabilities such as CAP_CHOWN. A container running uid 0 with cap_drop ALL does
not fully model that. The root lane should therefore start from cap_drop ALL as
the safer baseline, then add only the minimal capabilities required by a
specific metadata assertion. Expected candidates are CAP_CHOWN for owner/group
preservation and possibly CAP_FOWNER if chmod/chown edge cases require it.
Do not grant broad capability sets merely because the lane is called "root".

Rejected approaches
-------------------

1. "Just run current tests in Docker."
   Rejected because it does not prove host refusal, harness posture, docs
   safety, noexec tmp compatibility, distro matrix behavior, or root-specific
   metadata preservation.

2. Compose as the default.
   Rejected because there are no service dependencies. Compose would add config
   surface without improving isolation.

3. Read-write repo bind mount.
   Rejected because it weakens the guarantee that tests cannot touch the host
   checkout.

4. Host-runnable tests with a documented preference for containers.
   Rejected because "containerized tests only" is a hard rule.

5. Runtime network access.
   Rejected because tests do not need it. Package downloads belong to image
   build, not test execution.

6. Mounting the host Docker socket for "more realistic" Docker cleanup tests.
   Rejected because the cleanup script can be tested with fake Docker command
   fixtures. A host socket would turn tests into real host integration work and
   would violate the core safety boundary.

7. Floating base image tags such as stable, latest, or rolling testing.
   Rejected because they make failures hard to attribute. Use named releases
   plus digests; add new named releases deliberately.

8. Running the broad suite as root.
   Rejected because root masks permission mistakes. Use a narrow root lane only
   for owner/group preservation behavior.

Implementation sequence
-----------------------

Phase 1: harness and refusal

Pledge item:

- [ ] I pledge to implement Phase 1 in small slices: first the minimal image,
      then the wrapper, then the guard, then README verification. I will not
      split the whole test suite or add CI in this phase.

Why first:

No later safety claim matters while host-runnable tests remain the normal path.

Outcome:

The only supported test command is containerized, and existing tests run from a
copied image context without runtime host access.

Priority tiers
--------------

Tier 0 is the minimum acceptable change for "containerized tests only":

- host test entrypoints refuse execution
- make test runs tests inside a copied-context container image
- no runtime network
- no Docker socket
- no repo bind mount in default test lane
- broad lane is non-root
- README verification points to make test

Tier 1 is the minimum acceptable safety harness:

- noexec tmpfs after fake Docker refactor
- posture self-check inside the container
- wrapper posture tests
- .dockerignore/build-context tests
- docs contract checks
- exact staged install/uninstall file-set checks

Tier 2 is completeness:

- split monolithic tests by contract
- Debian/Ubuntu matrix
- narrow root metadata lane
- CI workflow, if wanted
- optional security/staleness reporting for pinned bases

Do not present Tier 0 as complete. It is only the first gate that makes further
test work safe to run.

Open questions to decide during implementation
----------------------------------------------

1. Whether to support podman officially.
   Use DOCKER_CLI?=docker and keep commands close to OCI-compatible flags, but
   treat Docker as the supported default until podman is tested.

2. Whether to keep a debug shell.
   Useful, but it must not become a backdoor host test lane. It should run in
   the same image and containment profile, with a read-only repo view if mounted.

3. Whether CI should run all distro images on every push.
   Full matrix is better safety; Debian 12 only is faster. A pragmatic split is
   Debian 12 locally, Debian 12 plus Ubuntu 24.04 in CI.

4. Whether to adopt snapshot package repositories.
   Start without snapshots so distro tool drift remains visible. Move to
   snapshots only if CI reproducibility becomes more important than drift
   detection.

Definition of done
------------------

The test system is done when:

- Running ./tests/run.sh on the host fails with the container-only message.
- make test builds a pinned test image and runs all tests with no runtime
  network, no user-provided host path mounts, no Docker socket, read-only
  rootfs, and non-root user.
- The broad suite writes only to container-local tmpfs.
- A posture test fails if the runner loses the critical containment flags.
- README verification points only to containerized test commands.
- The suite proves installer, uninstaller, policy detection, Docker JSON merge,
  Docker cleanup safety, docs, and harness posture as explicit contracts.
